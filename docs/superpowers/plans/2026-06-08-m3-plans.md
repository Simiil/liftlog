# M3 — Plans Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Templates make session start instant — a Plans area (plan → day templates → reorderable exercise editor with targets), a template→session **snapshot** copy, Home quick-start chips, and target hints in the logging UI (05-roadmap M3).

**Architecture:** MVVM + UDF on the M1 data layer / M2 logging flow. One ViewModel per screen exposing an immutable `StateFlow<XyzUiState>`; ViewModels depend only on `domain/repository` interfaces (never DAOs/`Context`); reads are `Flow`, writes are `suspend`. The structural invariant (02-data-spec §1): **sessions snapshot template content, never reference it** — starting a session copies template rows into `session_exercises` (order + targets frozen), so editing/deleting a template never rewrites history.

**Tech Stack:** Jetpack Compose + Material 3, Navigation Compose (type-safe `@Serializable` routes), Hilt (`hiltViewModel()`), Coroutines/Flow, the M1 Room layer. New dependency: **`sh.calvin.reorderable`** (Compose-native drag-to-reorder; owner-approved 2026-06-08 — see flagged decision #1). Tests: JUnit + Turbine + coroutines-test (JVM, local) for ViewModels/repos/pure logic; instrumented Room tests (CI, locally compiled + runnable on the API-34 path) for new DAO queries + snapshot isolation; one Compose UI test (CI) for the 3-tap cold-start→template→log path.

## Decisions flagged for owner review (milestone gate)

Reasoned defaults where the docs leave room; flagged per CLAUDE.md rather than blocking. Reject any at the gate and the change is localized.

1. **Reorder = `sh.calvin.reorderable` (third-party).** 03-ux-spec §6 specifies a drag handle; Compose has no first-party reorderable, so a custom gesture impl or a library is required. **Owner chose the maintained library (2026-06-08).** It's Compose-native, Apache-2.0, small, actively maintained, integrates with `LazyColumn`. Justified under CLAUDE.md constraint #5 (prefer Jetpack/well-maintained; no Jetpack equivalent exists). Confined to the Template editor screen.
2. **Screen hierarchy = 3 screens** (Plans list → Plan detail → Template editor), literal to 03-ux-spec §6 ("plan list → templates → editor"). The §2 nav map collapses this to two boxes; §6 text is authoritative. Each screen stays small and independently testable.
3. **`createPlan`/`createDayTemplate`/`addExerciseToTemplate` append at `max(position)+1`** (was hardcoded `0` for plans — M1 forward note). First item gets position 0. New items go to the end (stable, predictable).
4. **Home chips = day templates of the most-recently-used plan, else the first plan by position.** "Most recently used" = the plan owning the template of the latest started session (DAO join). With zero plans → no chips, just the empty-session card (current behavior preserved).
5. **`startSessionFromTemplate` reuses the single-active-session guard.** If a session is already live it throws (same as `startEmptySession`); Home guards by routing chip taps to *resume* when a session is live (mirrors the existing `startOrResume`). The copy is transactional: session + all snapshot `session_exercises` in one `Transactor.immediate`.
6. **Plan-level and day-level reorder UI is deferred** (not in M3 exit criteria; only the *exercise list* is "reorderable" per §6). Positions are still assigned `max+1` so a future reorder UI is a pure add. Flagged.
7. **Target editing UI = inline, no dialogs.** Tapping an editor row expands it to show small `−/+` steppers for target sets and rep-min/rep-max; "0"/cleared stores `null` (no target). Naming a plan/day uses a small text dialog (acceptable — Plans editing is not the logging hot path; the "one dialog" rule in §4.4 is about the *session* screen).
8. **Rep-range hint formatting** (`Targets.repRangeHint`): both null → `null` (stepper shows "reps"); `min==max` or one side null → the single number (`"8"`); else `"8–12"` (en-dash `–`). Pure + unit-tested.
9. **Plans list shows plan names only** (no per-plan day-count badge) to avoid an N+1 flow/extra query; day counts are visible one tap deeper on Plan detail. Flagged as a deliberate omission.
10. **`reorderTemplateExercises(orderedIds)` persists positions by list index** in one transaction; the editor calls it on drag settle. Cheap (a handful of `UPDATE`s), deterministic, JVM-testable.
