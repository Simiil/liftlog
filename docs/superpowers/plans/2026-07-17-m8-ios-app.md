> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** LiftLog runs on real iPhones via TestFlight — an `iosApp/` Xcode shell around the existing `LiftLogKit.framework`, real implementations for the deferred iOS seams, idiom polish, a macOS CI lane, and a documented signing/upload pipeline.

**Architecture:** M7 left the entire app (UI included) compiling in `commonMain` with 195 common tests green on the iOS simulator; every platform seam has an iOS actual, some deliberately minimal. M8 adds the thin Swift shell (the official KMP template shape: one `UIViewControllerRepresentable` around `MainViewController()`, an `embedAndSignAppleFrameworkForXcode` build phase), replaces the minimal actuals with real ones (document picker, bundle version, DB dispatcher), and verifies the M7 debt-list parity items on the simulator. Spec: [2026-07-15-ios-port-design.md](../specs/2026-07-15-ios-port-design.md); debt list: issue #47.

**Tech Stack:** Xcode 26.6 (installed, iOS 26.5 simulator runtime present), Kotlin 2.3.21 / CMP 1.11.1 (pinned since M7), `LiftLogKit` static framework (`isStatic = true`, both iOS targets), SwiftUI shell (~25 lines of Swift total), GitHub Actions `macos-15` (repo is public → macOS minutes are free).

- [ ] **Step 1: Implement `DocumentPickers.ios.kt`.** Shape (executor adapts exact interop spellings against the platform stubs — acceptance is compile + on-sim behavior):

```kotlin
// Export: LiftLog writes the JSON to a temp file itself, then hands it to the picker
// for the user to place ("forExporting" moves the file). The common SettingsScreen flow
// calls the returned lambda with a suggested filename, expects onResult(handle-to-write).
// On iOS we invert: produce the temp file URL as the DocumentHandle; SettingsViewModel's
// export writes to it via DocumentIo BEFORE the picker is presented — order verified in Step 3.
@Composable
actual fun rememberCreateDocumentLauncher(
    mimeType: String,
    onResult: (DocumentHandle?) -> Unit,
): (String) -> Unit {
    return remember(mimeType, onResult) {
        { suggestedName ->
            val tmp = NSURL.fileURLWithPath(NSTemporaryDirectory() + suggestedName)
            onResult(DocumentHandle(tmp))
            // Presentation of UIDocumentPickerViewController(forExporting = listOf(tmp),
            // asCopy = true) happens after the ViewModel finishes writing — see
            // IosDocumentExportPresenter below wired through a completion callback.
        }
    }
}
```
**Design note the executor must resolve concretely (read `SettingsScreen.kt` + `SettingsViewModel.kt` first):** on Android, CreateDocument returns a URI *then* the ViewModel writes to it. On iOS the file must exist *before* `forExporting` presentation. The clean adaptation: the iOS launcher creates the temp `DocumentHandle`, invokes `onResult` (ViewModel writes via `DocumentIo`), and presents the export picker when the write completes. If the common flow offers no completion hook, add one minimally: `SettingsViewModel.export` already ends by setting `SettingsMessage.EXPORTED` — the iOS actual can observe nothing from there, so instead present the picker inside `IosDocumentIo.writeText` completion for temp-export handles… **STOP: do not improvise beyond this** — if the seam needs a common-side signature change (e.g. `onWritten: () -> Unit` on the launcher), report DONE_WITH_CONCERNS in your task report and propose it explicitly; the reviewer decides. Import direction is straightforward: `forOpeningContentTypes` → delegate `didPickDocumentsAtURLs` → `onResult(DocumentHandle(url))`.
Presentation host: top view controller via `UIApplication.sharedApplication.keyWindow?.rootViewController` (walk `presentedViewController` chain); delegate object must be retained (store in a holder tied to the composition via `remember`).
